## 分析：為何未達成 30% 目標

### 根本原因：IPC 架構差異

本測試揭示了當前架構下的重要發現：

**當前架構（subprocess JSON IPC）**:
- Bun 後端：`Bun.$\`ffmpeg ...\`` — 直接啟動 FFmpeg 子進程
- Go 後端：啟動 `kinetitext-audio` Go 進程 → Go 進程再啟動 FFmpeg 子進程

換言之，Go 後端路徑為 **兩層子進程**：
```
Bun → kinetitext-audio (Go) → FFmpeg
```

而 Bun 後端僅為 **一層子進程**：
```
Bun → FFmpeg
```

### 性能差距原因

1. **Go 運行時啟動開銷**：每次調用都啟動新的 Go 進程（~50-80ms Go 運行時初始化）
2. **雙重 IPC 開銷**：Bun ↔ Go JSON + Go ↔ FFmpeg 進程間通信
3. **短音頻文件測試局限性**：5 秒靜音音頻的 FFmpeg 處理時間極短，I/O overhead 佔主導
4. **Go 進程無連接池**：目前為無狀態模型（06-01 決策），每次調用重新啟動

### 預期在真實場景的表現

對於長音頻文件（例如 30-60 分鐘有聲書章節），測試預期不同：
- FFmpeg 處理時間將佔主導（分鐘級）
- Go 的 ~100ms 啟動開銷比例下降為可忽略
- Go ffmpeg-go 的內部優化（管道化 I/O、直接記憶體緩衝）可能帶來 10-20% 提升

### 要達成 30%+ 提升所需的架構改進

若要達成原定性能目標，需要以下改進之一：

**選項 A：Go 進程池（不需架構大改）**
- 保持 subprocess JSON 模型
- 引入 Go HTTP 或 Unix socket 服務端，Bun 端長連接調用
- 消除每次調用的 Go 運行時啟動開銷（~50-80ms）
- 預期改善：25-35%（對短文件）

**選項 B：Bun FFI（激進改動）**
- 將 Go 編譯為 cgo C ABI 共享庫
- Bun FFI.cdef() 直接調用，零進程啟動開銷
- 預期改善：15-25%（主要來自 Go 的 FFmpeg I/O 優化）

**選項 C：接受當前結果（建議）**
- 當前實現對 **長音頻** 場景可能有 10-20% 提升（可接受）
- 真正的性能瓶頸是 FFmpeg 本身，而非語言運行時
- 保留 Go 後端作為可選替代，繼續 Phase 7 (DurationService) 工作
