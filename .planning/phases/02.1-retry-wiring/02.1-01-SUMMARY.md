# Phase 2.1 Plan 01: Wire AudioErrorClassifier into RetryService — Summary

**One-liner:** FFmpeg-aware error classification wired into RetryService via constructor DI, enabling intelligent retry decisions for codec/memory/pipe errors.

## What Was Done

AudioErrorClassifier was previously defined but never used during retry decisions. This gap closure wires the classifier into RetryService through optional constructor injection, and updates both AudioConvertService and AudioMergeService to inject AudioErrorClassifier when creating their RetryService instances.

## Changes Made

### RetryService (`src/core/services/RetryService.ts`)

Constructor signature changed from:
```typescript
constructor(config: RetryConfig, logger?: pino.Logger)
```
to:
```typescript
constructor(config: RetryConfig, errorClassifier?: ErrorClassifier, logger?: pino.Logger)
```

Uses `errorClassifier ?? new ErrorClassifier()` so all existing callers remain backward compatible.

### AudioConvertService (`src/core/services/AudioConvertService.ts`)

```typescript
// Before
this.retryService = deps.retryService ?? new RetryService(retryConfig)

// After
this.retryService = deps.retryService ?? new RetryService(retryConfig, new AudioErrorClassifier())
```

### AudioMergeService (`src/core/services/AudioMergeService.ts`)

Same pattern as AudioConvertService.

### RetryService.test.ts

Added "Dependency Injection" describe block with two tests:
- `accepts custom errorClassifier in constructor` — verifies AudioErrorClassifier injection works
- `uses base ErrorClassifier by default when no classifier provided` — verifies backward compatibility
